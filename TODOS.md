## P1 — SRS Unit Tests

**What:** Unit tests for `useSRS.js` — interval progression, `getDue()` logic, localStorage write/fail scenarios.

**Why:** SRS bugs cause users to study the wrong words at the wrong time. Learning tools have
zero tolerance for correctness bugs. The algorithm (interval ×2.5, reset to 1 day, getDue filter)
is exactly the kind of logic that's trivial to test and easy to get subtly wrong.

**Pros:** Catches interval regression before it ships. Confidence to refactor later.

**Cons:** Requires adding a test runner (vitest) — currently no test setup exists.

**Context:** `useSRS.js` will live at `src/composables/useSRS.js`. It's a pure function composable
(localStorage side effects are wrappable). Add vitest + happy-dom for browser API simulation.

**Effort:** S → with CC: S (human ~4h / CC ~15min)
**Priority:** P1
**Depends on:** useSRS.js completed

## P2 — PWA Push Notifications

**What:** Service Worker + Web Push API for SRS review reminders: "你有 5 個單字要複習！"

**Why:** SRS without a re-entry hook is a gym with no front door. Users forget to open the
app the next day. PWA manifest (in this iteration) enables home screen install; push
notifications close the loop.

**Pros:** The highest-leverage retention feature after SRS itself. One notification = user
returns = SRS works.

**Cons:** Requires Service Worker + Cloudflare Workers push dispatcher. More infra than current
stack. Push permission UX is tricky (don't ask too early).

**Context:** Start with "add to home screen" (PWA manifest, already in this plan). Then add
Service Worker with push. Cloudflare Workers can dispatch push payloads via Web Push API.
Ask for permission after the user completes their first review session.

**Effort:** M → with CC: S (human ~1 week / CC ~30min)
**Priority:** P2
**Depends on:** PWA manifest shipped, SRS shipped

## P2 — Vocabulary Progress Dashboard

**What:** Cross-video learning statistics — total words learned, tier distribution (A1/A2/B1/B2/C1+),
SRS review accuracy rate over time.

**Why:** First feature that makes users feel "I'm actually improving." Progress visibility is
a proven retention driver. Currently users have zero feedback on their improvement.

**Pros:** Gives users a reason to keep coming back beyond today's video. "From A2 to B1 in 2 weeks."

**Cons:** Requires aggregating SRS localStorage data. Needs a dashboard page/component.

**Context:** All data is available in localStorage after SRS ships (`jolike_srs_*` keys).
A simple dashboard at `/progress/` showing tier breakdown + streak is 100% client-side.
Depends on SRS being used long enough to accumulate meaningful data.

**Effort:** M → with CC: S (human ~1 week / CC ~20min)
**Priority:** P2
**Depends on:** SRS completed, users have used SRS for 1+ weeks
